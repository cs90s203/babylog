# 寶寶日誌 — 版本記錄

版號規則跟 JPL（日語學習 App）主 dev 對話一致：`MAJOR.MINOR.PATCH`——新功能升 MINOR、
架構級的重大改動升 MAJOR、bug 修正/小改善升 PATCH。版本號顯示在設定頁最下方。

## v2.17.1 — 2026-07-04

**UI調整：珊瑚色統一常駐導覽列/月曆熱力圖；單日時間軸新增密集度熱力色帶**

- 常駐導覽列（首頁/統計/紀錄/設定）選中時的顏色，跟月曆熱力圖圓圈的顏色，都改成跟統計頁
  「喝奶次數」長條圖一致的珊瑚色（`#FF8C6B`）。其他地方（FAB圓鈕、主要按鈕、時間軸「現在」
  線、✓準標記）維持原本的金色，App 現在有兩種強調色分工。
- 單日展開／比較模式的時間軸，新增密集度熱力色帶：每10分鐘一格，用當下紀錄類型的顏色（喝奶
  米色系、排便棕色、尿尿藍色、刷牙綠色）畫一條貫穿整個時間軸寬度的色帶，同一時段紀錄越密集
  顏色越深；色帶上下邊界會淡出（用同一個顏色只調整透明度，不會換色），同一時段有多種類型時
  會自然疊加混色。
- 修正過程中的一個 bug：色帶一開始只做了剛好10分鐘高，淡出效果被壓縮在幾個像素內，看起來像
  一條死板的實心線而不是柔和的光暈——改成每個色帶畫得比實際時段更高（上下往外延伸），淡出
  才有足夠空間呈現，也讓相鄰時段的色帶能自然融合、更像真正的熱力圖。

## v2.17.0 — 2026-07-04

**紀錄頁大改版：月曆形式、單日展開、多日比較；統計頁新增假日規律洞察**

- 紀錄頁不再是流水帳列表，改成月曆形式，從第一筆紀錄的月份開始算，只能在「最早月」到
  「當月」之間翻頁。
- 每天日期數字的背景是一個熱力圖圓形色塊——顏色深淺依照當天紀錄次數的多寡，沒有紀錄的日子
  就沒有色塊；一個視覺元素同時講完「有沒有紀錄」跟「那天忙不忙」兩件事。
- 點一個有紀錄的日子，下方會展開那一天24小時的完整時間軸（照實際時鐘時間排列，不壓縮空白
  時段）。
- 新增「📊 比較」模式：最多選4天，並排比較，每一天都對準同一個時鐘刻度（例如四天的凌晨3點
  會排在同一個高度），所以能直接看出「這幾天同一個時段在幹嘛」。選的天數越多，每欄會自動
  縮減內容（完整標籤 → 只留奶量 → 只留emoji），避免擠成一團。
- 統計頁「平均奶量/日」旁邊新增一欄「假日規律」——比較週末 vs 平日「第一餐」的平均時間，看
  假日通常晚起（或早起）幾分鐘；資料量還不夠時會先顯示「資料不足」。

## v2.16.4 — 2026-07-04

**新增：預測「準確」的標記，順便解決文字互相干涉的問題**

- 誤差在 ±15分鐘以內算「預測成功」，實際那筆紀錄旁邊會改顯示綠色「✓準」標記，後面接奶量
  誤差（例如「✓準 +20ml」）；如果時間不是完全吻合、但還在±15分鐘內，也會補上分鐘數（例如
  「✓準 +8m -10ml」）。
- 這個門檻也順便解決了你在 08:30 那筆看到的文字互相干涉的問題：預測「準」的時候，代表預測
  時間點跟實際時間點本來就幾乎在同一個位置，兩邊各畫一次等於是重複資訊——現在準確的情況只
  畫「✓準」這一塊，不再另外畫容易疊在一起的虛線預測標記；只有誤差超過±15分鐘、兩個時間點
  在時間軸上有明顯差距時，才會分開顯示「誤差」文字跟另一個位置的虛線標記。

## v2.16.3 — 2026-07-04

**修正：預測時間的虛線標記被實際紀錄的色塊擋住看不到**

- 找到真正原因了：v2.16.2 新增的「預測時間虛線標記」，原本是貼齊在時間軸左邊（跟事件色塊
  同一欄，`left:軸線+14px` 到 `right:4px`）——只要那個預測時間點剛好靠近某一列有色塊的實際
  紀錄，色塊本身不透明的背景就會直接蓋在虛線標記上面，整個看不到，只剩下沒有紀錄的空白列
  才看得到（例如畫面最下面靠近「現在」的那一個）。
- 改成跟「誤差 +/-幾分鐘」文字一樣，貼在時間軸最右邊那一欄（跟色塊完全不同欄位，不會互相
  遮住），現在每一個預測時間點都能正常顯示在時間軸上該有的位置。

## v2.16.2 — 2026-07-04

**修正：預測時間標籤真的要對到時間軸上對的位置，不是只有間距問題**

- v2.16.1 只解決了「文字有沒有垂直置中」，但你反應的其實是另一個問題：那段文字寫著
  「預測 12:10」，卻是貼在實際那筆餵奶紀錄（例如09:34）的那一列旁邊——時間軸上09:34那個
  高度跟「12:10」完全對不上，看起來就像亂放的。
- 這次改成兩塊分開處理：① 實際那筆紀錄旁邊只留下「誤差 +幾分鐘、+幾ml」這種差距數字，不再
  寫預測的時鐘時間；② 另外在時間軸上「預測的那個時間點」本身的位置，畫一個虛線的小圓點跟
  「┄ 預測 12:10・100ml」文字——這樣預測點才會真的出現在時間軸上12:10的地方，跟實際發生的
  09:34分開顯示，一眼就能看出差了多久、差在哪裡。
- 如果預測的時間剛好落在時間軸上已經摺疊收起來的「⋯ 無紀錄」空白段裡，這個預測點會先隱藏
  （摺疊段本來就沒有實際比例的位置可以畫），點開那段展開後就會正常顯示。

## v2.15.1 — 2026-07-04

**緊急 Bug 修正：整個 App 打開沒畫面**

- v2.15.0 把整個 App 弄壞了——`predict.js` 跟 `views.js` 兩個檔案都宣告了同一個
  `const EARLY_HOUR`，這個專案沒有用 ES module（純 `<script>` 標籤），同一份網頁裡所有
  `<script>` 標籤其實共用同一個全域作用域，同名的 `const` 宣告兩次會直接丟出
  `SyntaxError: Identifier 'EARLY_HOUR' has already been declared`，整個 `views.js`
  當場停止解析，`render()` 等函式全部沒定義，App 開起來完全空白。
- 修法：把 `predict.js` 裡的那個常數改名成 `PREDICT_EARLY_HOUR`，避開命名衝突；順便掃過
  所有 js 檔案的頂層變數/函式命名，確認沒有其他重複。
- 這次的測試疏漏：先前驗證 v2.15.0 時，我是直接呼叫 `predictNextFeed()` 這個函式本身
  （它在 `predict.js`，比 `views.js`先載入，衝突發生前就已經定義好），沒有實際確認整個
  App 畫面有沒有正常渲染出來，才會沒抓到這個問題。以後測試會加一道「確認畫面真的有內容、
  沒有 JS 錯誤」的檢查，不能只測試單一函式。

## v2.15.0 — 2026-07-04

**新功能：預計下一餐演算法大升級——早中晚分段、首末餐錨點、資料有效性過濾**

1. **資料有效性**：算「平常模式」（各時段中位數、首末餐時間、平均奶量/次數）的時候，
   現在會排除「今天」（永遠不完整）跟「第一次使用只有半天紀錄」的那個第一天（沿用統計頁
   `validStatsDays()` 同一套規則：第一天要在早上 6 點前就開始記才算數）。決定「下一餐」還是
   照樣用今天實際最後一餐當基準，只有拿來訓練歷史模式的資料會過濾。
2. **早中晚四段分類**（取代原本的白天/夜間二分法）：早上（5-11）、下午（11-17）、晚上
   （17-23）、夜間（23-5 或跨到隔天），各自獨立算中位數間隔。預測時看現在落在哪個時段，
   用對應的中位數；如果已經超過該時段平常的間隔，或是已經過了平常的最後一餐時間（見下），
   會切換成用夜間中位數，邏輯是原本「超過白天間隔就切夜間」的推廣版，不是只有日夜兩種。
3. **首末餐錨點**：從有效資料裡算出「平常第一餐大概幾點」「平常最後一餐大概幾點」。今天
   還沒有任何一餐紀錄時（例如清晨剛醒），不再從「昨晚最後一餐＋夜間間隔」去外推（容易推
   過頭），改成直接預測「今天的平常第一餐時間」；平常最後一餐時間也拿來輔助判斷該不該提早
   切換成夜間模式。
4. **平均奶量／平均餐數**：用同一批有效資料算出「平均一天喝幾 ml」「平均一天餵幾次」，
   先算出來存在回傳結果裡備用（`avgMlPerDay`／`avgFeedsPerDay`），這次沒有直接拿來改變
   預測時間本身，也還沒有在畫面上顯示，之後想用再加。
5. 驗證：模擬「第一天下午才開始記錄（排除）＋連續 4 個完整天（規律作息）＋今天還沒餵過」
   的情境，確認正確排除無效天數、正確算出平均奶量/次數、正確用「平常第一餐時間」預測今天
   第一餐，而不是死套間隔外推；`bucketOf()` 的早/中/晚/夜分類邏輯也逐一驗證過對應正確的
   時段。

## v2.14.1 — 2026-07-03

**新功能：平均值也排除「還沒開始追蹤」的天數，只有紀錄完整的第一天才算數**

- 補充 v2.14.0：如果家人是某週三才第一次開始用這個 App 記錄，週一、週二完全沒紀錄的那幾天
  不是「寶寶那天很乖」，是根本還沒開始追蹤，這種天數也要從平均值的分母排除，不只排除還沒

  不完整，一併排除。
- 修正一個實作疏漏：只調整分母（排除的天數）還不夠，分子（排便/尿尿/奶量的加總）也要
  同步排除那些被排除天數的紀錄，不然分子分母對不起來，算出來的平均反而會比正確值更高。
- 驗證：模擬「週三下午才開始記錄」的情境，確認週一週二（尚未追蹤）、週三（下午才開始，不算
  完整）都被排除在分母跟分子之外，平均排便次數算出來的數字跟手算結果一致；另外驗證「當天
  凌晨 2 點就開始記錄」的情境，那一天正確被視為完整、納入計算。

## v2.13.1 — 2026-07-03

**Bug 修正：成長彈窗填完數字後，iOS 畫面沒有自動縮放回正常大小**

- 根因：體重/身高/頭圍這三個輸入框字體是 14px，低於 iOS Safari 判斷是否要自動放大畫面
  的門檻（16px），點下去打字時畫面會被系統自動 zoom in；理論上失焦時應該自動縮回去，但
  這次失焦的原因是「彈窗被整個重新渲染關閉」而不是使用者手動點旁邊，Safari 常常抓不準這種
  時機，導致縮放卡住沒有跳回來。
- 修法：不改這三個輸入框的外觀，改成在儲存/刪除/取消關閉成長彈窗的當下，用一個常見技巧
  強制把畫面縮放拉回 100%——暫時把網頁的 viewport 設定加上 `maximum-scale=1.0` 逼 Safari
  立刻歸位，350ms 後再把設定改回原本沒有限制的樣子，使用者之後還是可以正常手動雙指縮放
  整個 App。
- 驗證：確認儲存成長紀錄的當下，viewport 設定被正確加上縮放限制；350ms 後正確恢復成原本
  的設定內容。

  （純小數，可以超過 24），實際換算靠真正的 `Date` 物件運算，跨天自然正確，不需要特別處理
  午夜。空檔壓縮功能（≥3 小時無紀錄自動收合）邏輯完全不變，只是套用在新座標上。
- 拖曳事件調整時間會抖動、往左拖會消失、拖曳一開始頁面會跳回頂端——三個症狀追到同一個根因：
  舊版拖曳時，手指每移動一像素就呼叫一次 `Store.updateEvent()`，觸發整個 App 重新渲染一次
  （`#root.innerHTML` 整個換掉）。畫面重繪那麼高頻，除了卡頓，捲動區域的 DOM 節點也被整個
  換成新的（重置捲動位置），拖曳用來換算座標的參考節點也可能抓到舊的/已卸載的節點，算出無效
  時間讓事件消失。修法：拖曳中只直接操作那顆事件標籤自己的 DOM 位置/文字（不寫入 Store、不
  觸發重繪），放開手指才真正寫入一次資料。額外加上：`render()` 現在會在整頁重繪前後保存/還原
  捲動位置，就算之後別的地方又不小心觸發重繪，也不會再無故跳回頂端。

## v2.1.0 — 2026-07-01

**新功能：下拉重新整理（純重整頁面，跟同步無關）**

- Firebase 版拿掉了舊版下拉同步手勢後，重新加回一個下拉重新整理功能——但這次純粹是重新載入
  整個頁面（`location.reload()`），不再嘗試觸發任何同步邏輯（Firestore 本來就即時同步，
  不需要靠這個手勢），單純給需要「重新開始乾淨狀態」時用（開發測試、或裝置的監聽器卡住時）
- 下拉時會有指示條浮現（「↓ 下拉重新整理」→ 拉超過門檻變「↓ 放開重新整理」），放開後短暫
  顯示「重新整理中…」再重新載入
- 指示條元素刻意掛在 `#root` 外面（直接接在 `body` 下），因為 `#root` 每次畫面重繪都會被
  整個換掉，如果指示條掛在裡面，手勢做到一半重繪就會把它砍掉重建
