# 寶寶日誌 — 版本記錄

版號規則跟 JPL（日語學習 App）主 dev 對話一致：`MAJOR.MINOR.PATCH`——新功能升 MINOR、
架構級的重大改動升 MAJOR、bug 修正/小改善升 PATCH。版本號顯示在設定頁最下方。

## v2.7.0 — 2026-07-03

**新功能：統計頁圖表可左右滑動切換週/月/年、顯示日期、點擊展開奶量細節**

1. 喝奶次數、奶量 ml、換尿布次數這三張圖表現在可以左右滑動切換上一週/上一月/上一年，
   本週/本月/本年三個分頁都適用。往前可以一路滑到最早有紀錄的那個週期，不能滑到未來。
   滑動方向比照 Apple 健康 App 的週圖表：往左滑＝看更早以前，往右滑＝往現在的方向。
2. 分頁按鈕下方新增一行小字副標題，顯示目前正在看的週期（例如「6/29（一）－7/5（日）」
   或「2026年6月」），滑動切換時可以清楚知道自己滑到哪裡了；本週模式下三張圖表的星期
   標籤下方也各自加上日期（例如「一」下面多一行「6/29」）。
3. 奶量 ml 圖表現在可以點一下柱子展開細節：本週模式點下去顯示當天母乳/配方各喝了多少
   ml；本月/本年模式下一根柱子代表一整週/一整月的加總，點下去就展開那一週/那個月的
   母乳/配方加總。
4. 驗證：模擬左右滑動確認可以正確切換週期、在最早紀錄的週期跟現在的週期都會卡住不能再
   滑過去；切換本週/本月/本年分頁會重置回目前週期並收起展開的細節；點擊奶量柱子在三種
   模式下都能顯示正確的日期/週次/月份標籤跟對應的母乳配方數字。

## v2.6.0 — 2026-07-03

**Bug 修正＋新功能：首頁數字午夜沒歸零、餵奶間隔排除夜間、排泄次數分開顯示**

1. 首頁的喝奶/ml/排便/尿尿數字過了本地午夜還沒歸零——根因是判斷「今天」的 `dayKey()` 用
   `toISOString()` 抓 UTC 日期，不是手機的本地日期。台灣是 UTC+8，UTC 日期要到本地時間早上
   8 點才會跨天，所以這幾個數字實際上是本地早上 8 點才歸零，不是午夜。改成用本地
   年/月/日（`getFullYear`/`getMonth`/`getDate`）判斷，午夜就會正確歸零。
2. 統計頁「平均餵奶間隔」現在會排除夜間睡眠的空檔，不然那段長間隔會把平均值拉高、不能反映
   白天實際的餵奶節奏。排除方式不是設固定時段（例如 22:00-06:00），而是動態判斷：只要兩次
   餵奶剛好跨到不同的日曆日，那個間隔就當作「夜間空檔」跳過不算，其餘同一天內的間隔正常
   計入平均。
3. 統計頁摘要卡片的「排泄次數」（排便+尿尿相加）改成分開顯示「排便次數」「尿尿次數」兩個
   獨立數字，比較貼近一般醫院衛教常見的分開記法（尿布濕度看奶量夠不夠、排便看腸胃狀況，
   兩者意義不同不適合加總）。
4. 驗證：模擬本地日期跨天的兩筆事件（UTC 同一天但本地不同天）現在會被正確判斷成不同天；
   刻意跨日的長間隔（模擬夜間空檔）確認被排除在平均值之外，只用同一天內的間隔算出精確的
   平均數字；摘要卡片正確顯示分開的排便/尿尿次數。

## v2.5.0 — 2026-07-02

**新功能：換尿布次數改成正確的「同一時間才算同一次」演算法，長條圖同時顯示三個數字**

- 原本的「尿布更換次數」圖表其實只是排便次數 + 尿尿次數簡單相加，同一次換尿布如果排便跟
  尿尿各記一筆，會被灌水算成 2 次。改成 `dedupeDiaperChanges()`：只有事件時間**完全相同**才算
  同一次換尿布，就算只差 1 分鐘也算兩次（不是用時間軸那種寬鬆的分堆間隔，因為使用習慣是
  「每次不一定一次上完，但只要不是同一時間就是不同片尿布」）。
- 統計頁摘要卡片的「排泄次數」維持原樣（單純排便+尿尿事件總數），不跟著改成換尿布次數——
  因為那張卡片的重點是紀錄大小便本身發生幾次，換尿布次數只在下面這張圖表看。
- 長條圖每根柱子上方同時印出三個數字，各自用對應顏色：換尿布次數（一般文字色）、排便次數
  （棕色）、尿尿次數（藍色），一眼就能同時看到「換了幾次」跟「大便/尿尿各幾次」，不用切換
  畫面。
- 驗證：同一分鐘記錄的排便+尿尿算 1 次換尿布；1 分鐘後的另一筆排便正確算成獨立的第 2 次；
  當天總計 3 次換尿布、2 次排便、2 次尿尿，數字完全對得上。

## v2.4.1 — 2026-07-02

**Bug 修正＋UI 實驗：時間軸拖曳誤觸、快速記錄按鈕搬到底部導覽列**

- 時間軸事件太容易在上下滑動畫面時被誤觸拖曳。原因是長按啟動拖曳的計時器（240ms）不會因為
  手指其實正在移動（代表是在滑頁面）就取消，時間一到還是會硬切成拖曳模式、劫走你的滑動手勢。
  修法：計時器延長到 740ms，並且加上「計時器觸發前如果手指已經明顯移動，就取消這次拖曳準備」
  的判斷，兩者一起大幅降低滑動誤觸的機率。真正進入拖曳模式的那一刻，被抓住的卡片會發光一下
  確認「選到了」。
- UI 實驗：首頁「快速記錄」大按鈕區塊暫時隱藏（程式碼保留一個開關，隨時可以切回來），改成
  三顆縮小、只留 emoji 的按鈕，用簡單的三角函數算出角度座標，弧形排列在底部導覽列中間「＋」
  按鈕的正上方，看起來像肉墊上方的三根腳趾。「＋」本身的功能（記錄成長）不變；排便/尿尿
  按鈕的長按補記時間手勢也照樣保留。
- 驗證：模擬「按下後立刻移動」的滑動手勢，確認不會誤觸拖曳；模擬「按住不動超過 740ms」的
  長按，確認正常進入拖曳模式並發光；新的三顆導覽列按鈕點擊/長按功能跟原本大按鈕行為一致。

## v2.3.8 — 2026-07-02

**Bug 修正：紀錄很密集的日子，時間軸整點數字疊在一起、順序顛倒**

- 根因：小時格線位置的計算依賴各叢集「被往下推了多少」這個數字，但這個數字不是遞增的——
  緊密相鄰的叢集可能需要推很多，下一叢如果剛好間隔比較自然就完全不用推，兩叢之間這個「推移
  量」會突然掉回去，導致格線 Y 座標跟著往回彈，畫面上就會看到後面的整點數字疊到前面的數字
  上、順序顛倒（例如 07 疊在 05 的位置）。紀錄越密集、叢集越多的日子越容易出現。
- 修法：改成用每一叢實際落點的位置（這個位置本身保證只會往下遞增，不會跳動）當作錨點，
  在錨點之間用直線內插計算格線位置，而不是套用單一叢集的推移量做外推。用一堆錨點連成的線，
  只要每個錨點本身是遞增的，整條線就一定是遞增的，不會再出現彈回去的情況。
- 驗證：用一天 25 筆、幾乎每小時都有紀錄的高密度資料重現原本的 bug，確認修法後所有整點
  數字（00~15+）跟叢集本身的位置都嚴格遞增、不再重疊或顛倒；也重新驗證了拖曳事件調時間的
  功能沒有被這次改動影響，放開後的時間跟畫面上顯示的時間仍然完全一致。

## v2.3.7 — 2026-07-02

**Bug 修正：一開始拖曳事件，時間軸格子馬上跳一下**

- 根因：時間軸的叢集分組（同時段的多筆事件要不要疊在一起、位置在哪）計算，先前只要
  `dragId` 被設定（長按滿 240ms 正式進入拖曳）就會把「正在被拖曳的那筆」整個從分組資料裡
  剔除，改用獨立的浮動框顯示。如果它原本跟其他事件同屬一叢，抽走之後那叢的平均時間、要不要
  疊字縮寫都會重新計算一次，畫面上就會看到那叢（甚至後面被推開的叢）瞬間跳位——即使手指
  根本還沒移動。
- 修法：叢集分組的計算改成永遠用完整事件清單（含被拖曳中的那筆，它在 Store 裡的原始時間本來
  就沒變，只有畫面上的浮動框跟著手指走），只在真正畫出每張卡片的時候跳過被拖曳那張本身
  （因為浮動框已經顯示它了）。這樣同一叢裡其他事件、時間軸格線的位置完全不受拖曳影響；如果
  被拖曳的是「落單」一筆（原本自己一叢），那叢的點/時間標籤也會正確整個隱藏，不會留下孤兒
  標記。
- 驗證：3 筆事件同屬一叢時，拖曳其中一筆，剩下兩筆的位置在拖曳開始瞬間完全不變；落單一筆
  的事件被拖曳時，原本位置的點跟時間標籤正確消失，只剩浮動框跟著手指。

## v2.3.2 — 2026-07-02

**Bug 修正＋優化：拖曳時間對不上時間軸、整點文字被蓋住、時間/ml 調整方式、底部彈窗手勢**

- 拖曳時間軸事件時，黃色的即時時間標籤跟畫面上的時間軸格線對不上，放開後套用的時間也是錯的
  ——根因是上一版（v2.3.1）新增的 `yOfAdjusted()`（讓格線／現在線跟著被推開的事件群組一起
  往下位移）只套用在畫面繪製上，拖曳用的「螢幕座標→時間」反算函式仍然用沒有位移過的舊算法，
  兩者因此各算各的。修法：拖曳改用對 `yOfAdjusted()` 做數值反解（5 分鐘解析度取樣＋二分搜尋
  內插）算出時間，跟畫面繪製共用同一套邏輯，永遠一致。
- 整點時間文字（例如「23:00」）跟事件本身的時間標籤共用同一欄位，彼此靠近時會視覺重疊。改成

## v2.3.0 — 2026-07-01

**新功能：時間軸重疊事件改成堆疊卡片，喝奶滑桿改成不觸發整頁重繪**

- 時間軸同一時刻的多筆事件，原本會換行擠在一起；現在改成像一疊卡片往右錯開堆疊，2 筆以上
  重疊時自動縮寫成「emoji ＋ 一個字」（排便＝便、尿尿＝尿，喝奶只留 emoji＋ml），保持可讀。

  滑桿也一併套用同樣邏輯，但那邊因為要寫入 Firestore，改成「拖曳中只更新畫面文字、放開

## v2.2.2 — 2026-07-01

**Bug 修正：編輯視窗調整任何數值時反覆「重新彈出」，滑桿完全沒辦法拖**

- 根因：彈出視窗（`.sheet`）套用了 CSS 的滑入動畫（`animation: slideUp`）。App 的畫面更新
  方式是每次狀態改變就整個 `#root` 重新產生一次 HTML 塞回去，所以彈窗本身也是每次都變成
  全新的 DOM 節點——而全新節點套用到 `.sheet` 這個 class，動畫就會重新從頭播一次。時間
  加減按鈕點一下觸發一次重繪、奶量滑桿用 `oninput`（每移動一點點就觸發一次）觸發更多次，
  於是每調一次數值，視窗就整個往上滑一次動畫，看起來就像「反覆重新彈出」，滑桿因為畫面一直
  在跳動也拖不動。
- 修法：記住目前開著的是「同一個彈窗」還是「剛打開的新彈窗」（比對彈窗種類 + 正在編輯的紀錄
  id），只有真正「剛打開」才播放滑入動畫，同一個彈窗裡面調整數值造成的重繪一律不重播動畫。
  用模擬操作驗證過：時間加減按鈕連續按 5 次、奶量滑桿連續拖 6 個位置，動畫都正確被抑制，
  數值也都正確即時更新；關閉後重新打開，動畫恢復正常播放。

## v2.2.1 — 2026-07-01

**Bug 修正：兩支手機登入後看不到同一份寶寶資料**

- 根因：裝置登入 Google 帳號後，即時監聽器只會把「遠端 Firestore 已有的資料」拉回本機
  （remote → local），但裝置在登入之前／登入當下就已經存在本機的資料，從來沒有被推上去過
  （local → remote 只有「登入後新增的紀錄」才會觸發 `Store._cloudPush`）。結果兩支手機各自
  的舊資料卡在自己身上，永遠不會同步給對方，看起來就像「沒連在同一個寶寶上」。
- 修法：登入成功那一刻，額外執行一次 `Sync._pushAllLocal()`，把當下本機所有 events/
  growth/settings 整批推上 Firestore（`merge:true`，重複執行也安全），確保不管這支手機
  是不是第一次登入、本機有沒有登入前就存在的資料，都會補齊上去。

## v2.2.0 — 2026-07-01

**改善：時間軸改成固定 24+3 小時捲動視窗，拖曳改成不觸發整頁重繪**

- 活動時間軸「今天」不再是「日曆上的今天（00:00–24:00）」，改成固定的「現在往前 24 小時
  ～ 往後 3 小時」捲動視窗。舊邏輯用「小時的第幾小時（0–24 的小數）」當座標，天生沒辦法
  跨午夜——過午夜看不到前一天的紀錄，拖曳也拉不過午夜線。新座標改用「距離視窗起點幾小時」

  午夜。空檔壓縮功能（≥3 小時無紀錄自動收合）邏輯完全不變，只是套用在新座標上。
- 拖曳事件調整時間會抖動、往左拖會消失、拖曳一開始頁面會跳回頂端——三個症狀追到同一個根因：
  舊版拖曳時，手指每移動一像素就呼叫一次 `Store.updateEvent()`，觸發整個 App 重新渲染一次
  （`#root.innerHTML` 整個換掉）。畫面重繪那麼高頻，除了卡頓，捲動區域的 DOM 節點也被整個
  換成新的（重置捲動位置），拖曳用來換算座標的參考節點也可能抓到舊的/已卸載的節點，算出無效
  時間讓事件消失。修法：拖曳中只直接操作那顆事件標籤自己的 DOM 位置/文字（不寫入 Store、不
  觸發重繪），放開手指才真正寫入一次資料。額外加上：`render()` 現在會在整頁重繪前後保存/還原
  捲動位置，就算之後別的地方又不小心觸發重繪，也不會再無故跳回頂端。
